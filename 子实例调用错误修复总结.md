# 子实例调用错误修复总结

## 问题描述

用户报告在并行调用多个子实例时出现以下错误：

1. **错误信息**：
   - `"Tool permission stream closed before response received"`
   - `"Stream closed"`
   - `"unhandled errors in a TaskGroup (1 sub-exception)"`

2. **错误现象**：
   - 子实例轮次较多或执行时间较长时报错
   - 一个子实例报错会导致其他并行子实例也失败
   - 看起来像"串行"失败，但实际是并行取消

## 根本原因分析

### 原因 1：异常传播到 Claude SDK

**问题代码位置**: `src/claude_instance_wrapper.py:164-180`

```python
# ❌ 错误的设计
try:
    query_result = await sub_agent.query_text(...)
except Exception as sub_error:
    logger.error(...)
    raise  # ← 重新抛出异常！
```

**为什么这样有问题**：
1. 内层的 `raise` 将异常抛给调用者（Claude SDK）
2. SDK 的 `asyncio.TaskGroup` 捕获到异常
3. TaskGroup 会取消所有其他并行任务
4. 导致连锁失败

**MCP 工具的正确设计**：
- ✅ 返回错误响应：`{"content": [...], "is_error": True}`
- ❌ 抛出异常（会触发 TaskGroup 取消）

### 原因 2：Claude SDK 内部超时限制

**发现过程**：
- 用户调试发现 `execute_instance` 函数正常返回结果
- 但 `stream_manager` 仍然报 TaskGroup 错误
- 分析会话记录发现时间线：
  - `21:18:22` - 子实例开始执行
  - `21:19:17` - SDK 返回 "Tool permission stream closed" (55 秒)
  - `21:19:19` - 子实例实际完成 (57 秒)

**真实原因**：
- Claude SDK 有约 **60 秒的内部超时限制**
- 当子实例执行时间超过 55-60 秒时，SDK 会主动关闭工具流
- 即使 `execute_instance` 返回正常结果，SDK 也已经触发了 TaskGroup 取消

## 完整修复方案

### 修复 1：移除内层异常重抛

**文件**: `src/claude_instance_wrapper.py`
**位置**: 第 169-175 行

**修改前**：
```python
try:
    query_result = await sub_agent.query_text(
        prompt,
        record_session=True,
        resume_session_id=resume_session_id
    )
except Exception as sub_error:
    logger.error(f"子实例 {instance_name} 查询失败: {sub_error}", exc_info=True)
    raise  # ← 删除这个块！
```

**修改后**：
```python
# 直接执行，不捕获异常（让外层的 except Exception as e 处理）
# 这样可以避免异常传播到 Claude SDK 的 TaskGroup，防止取消其他并行任务
query_result = await sub_agent.query_text(
    prompt,
    record_session=True,
    resume_session_id=resume_session_id
)
```

**效果**：
- 异常在外层 `except Exception as e:` (第 220 行) 被捕获
- 外层返回错误响应格式而不是抛出异常
- SDK 收到正常的工具响应，不会触发 TaskGroup 取消

### 修复 2：添加超时参数控制

**文件**: `src/claude_instance_wrapper.py`
**修改位置**:

1. **工具参数定义** (第 85-88 行)：
```python
"timeout": {
    "type": "integer",
    "description": "超时时间（秒），默认50秒（留出余量避免SDK超时）"
}
```

2. **参数提取** (第 126 行)：
```python
timeout = args.get("timeout", 50)  # 默认 50 秒，留出余量避免 SDK 超时
```

3. **超时控制实现** (第 170-195 行)：
```python
# 使用 asyncio.wait_for 强制超时控制
import asyncio
try:
    query_result = await asyncio.wait_for(
        sub_agent.query_text(
            prompt,
            record_session=True,
            resume_session_id=resume_session_id
        ),
        timeout=timeout
    )
except asyncio.TimeoutError:
    # 超时错误：返回清晰的错误信息，不抛出异常
    error_msg = (
        f"子实例 {instance_name} 执行超时（{timeout}秒）。\n"
        f"提示：可以通过 timeout 参数增加超时时间，或简化任务。"
    )
    logger.warning(error_msg)
    return {
        "content": [{
            "type": "text",
            "text": error_msg
        }],
        "is_error": True
    }
```

**效果**：
- 在 SDK 内部超时（~60秒）之前主动终止
- 返回清晰的超时错误信息
- 不会触发 SDK 的 TaskGroup 取消机制
- 用户可以通过 `timeout` 参数调整超时时间

### 为什么默认 50 秒？

- SDK 内部超时约 60 秒
- 设置 50 秒留出 10 秒余量
- 确保在 SDK 超时之前我们主动处理
- 避免 SDK 强制关闭流导致的 TaskGroup 取消

## 技术要点总结

### 1. MCP 工具设计原则
- **必须返回响应**：`{"content": [...], "is_error": True/False}`
- **不应抛出异常**：异常会触发 TaskGroup 取消
- **优雅处理错误**：在工具内部捕获异常并转换为错误响应

### 2. 异步异常处理
- 内层 `try-except-raise` 在异步环境下容易导致异常逃逸
- 应该让异常直接被外层捕获并转换为响应
- 避免异常传播到调用者（SDK）

### 3. 超时控制策略
- 使用 `asyncio.wait_for` 主动控制超时
- 设置比 SDK 超时更短的时间（留出余量）
- 超时时返回错误响应而不是抛出 `TimeoutError`

### 4. TaskGroup 行为
- Python 3.11+ 的 `asyncio.TaskGroup` 会在任何子任务失败时取消所有任务
- 这是设计行为，不是 bug
- 工具必须返回响应而不是抛出异常

## 常见问题

**Q1: 为什么一个子实例失败会影响其他子实例？**
A: Claude SDK 使用 `asyncio.TaskGroup` 并行执行工具，当任何工具抛出异常时，TaskGroup 会取消所有其他任务。修复后工具返回错误响应而不是抛出异常，避免了这个问题。

**Q2: 为什么看起来是"串行"失败？**
A: 实际是并行执行的，但因为 TaskGroup 在捕获异常后立即取消所有任务，其他工具几乎同时收到 "Stream closed" 错误，时间差只有几十毫秒，看起来像串行。

**Q3: 如果子实例确实需要很长时间怎么办？**
A: 通过 `timeout` 参数增加超时时间。例如设置 `timeout: 120` 允许执行 2 分钟。

**Q4: 超时时会话会被记录吗？**
A: 会的。超时时子实例的会话仍然会被完整记录，包括已执行的部分和超时错误信息。

**Q5: 修复后还会有异常吗？**
A: 工具级别的错误（超时、执行失败）会返回错误响应，不会抛出异常。只有严重的系统级错误（如内存不足）才会抛出异常。

## 总结

**核心修复**：
1. 删除内层 `try-except-raise` - 避免异常传播到 SDK
2. 添加 `timeout` 参数和 `asyncio.wait_for` - 主动超时控制

**关键原则**：
- MCP 工具应该返回错误，而不是抛出异常
- 在 SDK 超时之前主动处理
- 一个工具的错误不应影响其他并行工具

**修复效果**：
- ✅ 子实例调用更加健壮
- ✅ 并行执行互不影响
- ✅ 清晰的错误提示
- ✅ 可配置的超时控制
