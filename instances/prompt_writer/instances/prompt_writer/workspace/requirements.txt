# 开源情报搜索系统 - 完整需求文档

## 项目概述
构建一个三层智能体架构的开源情报搜索系统，使用 AI 协调和执行复杂的网络搜索任务，最终生成内容丰富、图文并茂的综合报告。

---

## 系统架构

### 三层智能体结构
1. **主协调系统** (Main Orchestrator Agent)
2. **搜索执行子智能体** (Search Subagent)
3. **写作综合子智能体** (Writing Synthesis Subagent)

---

## 第一层：主协调系统 (Main Orchestrator Agent)

### 核心职责
- 需求收集与理解
- 搜索战略规划
- 任务拆解与分配
- 动态策略调整
- 情报综合分析
- 最终报告协调

### 关键特征与要求

#### 1. 需求收集阶段 (Critical Phase)
**原则**: 反复询问，直到彻底理解用户需求

**收集策略**:
- 灵活适应不同场景，根据用户需求实际情况确定收集哪些信息
- 可能的维度包括但不限于：
  - 搜索目标和期待结果
  - 时间范围
  - 地域范围
  - 信息来源类型偏好
  - 深度和广度要求
  - 特殊约束或限制

**输出要求**:
- 将完整需求以文件形式保存
- 文件应结构化、清晰、完整
- 作为后续所有搜索活动的基准文档

#### 2. 战略规划能力 (Strategic Planning)
**核心定位**: 战略规划者，而非战术执行者

**职责范围**:
- ✅ 确定搜索什么（What to search）
- ✅ 期待什么结果（Expected outcomes）
- ✅ 搜索的大方向和路径（High-level strategy）
- ❌ 不管具体使用什么工具（由子智能体决定）
- ❌ 不直接执行搜索（只通过 Task 工具调用 subagent）

**战略要求**:
- **开放性**: 不局限于常规思路
- **创新性**: 能够找到通往答案的最佳路径
- **综合性**: 能够综合场景和需求
- **适应性**: 根据反馈动态调整策略

#### 3. 任务拆解原则 (Task Decomposition) ⚠️ 极其重要!
**核心原则**: 禁止将复杂任务整体扔给单个搜索子智能体

**拆解标准**:
- 单个子智能体的任务必须**单一、明确**
- 复杂任务需要**主系统自己拆解**成多个简单任务
- 每个子任务有清晰的输入和期待输出
- 避免"请帮我搜索关于XXX的所有信息"这种过于宽泛的指令

**示例对比**:
- ❌ 错误: "搜索关于人工智能发展的所有信息"
- ✅ 正确:
  - 任务1: "搜索2023-2024年AI领域的重大突破事件"
  - 任务2: "查找主要AI公司的最新产品发布"
  - 任务3: "收集AI监管政策的最新动态"
  - 任务4: "获取AI行业投资数据和趋势报告"

#### 4. 思辨能力 (Critical Thinking) 🧠 核心竞争力
**必须明确体现的能力**:

**推理能力**:
- 根据已有情报推测新的搜索方向
- 识别信息缺口和盲点
- 预判可能的信息来源

**关联能力**:
- 发现不同信息之间的联系
- 构建知识图谱和信息网络
- 识别矛盾和一致性

**质疑能力**:
- 对搜索结果进行批判性分析
- 评估信息可靠性和权威性
- 识别偏见和局限性

**调整能力**:
- 根据搜索结果动态修改策略
- 及时放弃无效路径
- 开拓新的探索方向

**发散能力**:
- 能够从核心需求发散到相关领域
- 探索非常规信息源
- 进行横向和纵向的信息拓展

#### 5. 并行调度能力 (Parallel Coordination)
**并发限制**: 最多同时运行 5 个搜索子智能体

**调度策略**:
- 优先启动关键路径任务
- 平衡深度搜索和广度覆盖
- 根据任务依赖关系合理排序
- 动态调整并行任务优先级

#### 6. 文件管理与情报综合
**文件读取**:
- 主动读取所有子智能体返回的研究成果文件
- 不依赖子智能体的口头报告

**综合分析**:
- 对所有搜索结果进行交叉验证
- 识别信息的重复、矛盾、补充关系
- 构建完整的情报图景

**质量评估**:
- 评估搜索完整性
- 识别信息缺口
- 决定是否需要补充搜索

#### 7. 与写作智能体的协作
**调用时机**:
- 认为已收集足够信息，可以综合成报告时

**指令明确性** (极其重要):
- 必须清晰告诉写作智能体：
  - 需要读取哪些文件（完整路径列表）
  - 每个文件如何处理
  - 提取哪些信息
  - 报告的结构要求
  - 重点强调的内容

**迭代能力**:
- 可以多次调用写作智能体进行修改
- 基于用户反馈调整报告
- 必要时重新启动部分搜索

#### 8. 探索性和随机性
**自由度边界**:
- 可以尝试非常规搜索路径
- 可以主动探索相关但非核心的领域
- 但不能偏离用户的核心需求

**随机性应用**:
- 在多个可行方案中随机选择
- 偶尔尝试"意外"的搜索角度
- 给予子智能体一定的自主权

## 第二层：搜索执行子智能体 (Search Subagent)

### 核心职责
- 接收主系统的搜索指令
- 灵活使用各种搜索工具
- 执行具体搜索任务
- 保存研究成果文件
- 返回文件路径

### 关键特征与要求

#### 1. 角色定位
**战术执行者**:
- 负责具体的搜索执行
- 对工具使用有完全控制权
- 可以根据实际情况灵活调整战术

**自主性**:
- 在主系统给定的任务范围内，有完全自主权
- 可以创造性地使用工具组合
- 不需要事事请示主系统

#### 2. 工具使用能力 (Tool Mastery)
**必须精通 5 个搜索服务器**:

**服务 1: searxng_search**
- 类型: 多引擎并行搜索
- 能力: 关键词优化、结果过滤排序、搜索策略执行
- 使用场景:
  - 初步搜索和广度覆盖
  - 补充搜索
  - 验证搜索
- 配置掌握: 引擎选择、语言设置、时间范围、安全级别
- 最佳实践:
  - 根据需求选择合适的搜索引擎组合
  - 优化关键词以获得更精准结果
  - 使用时间过滤器获取最新信息

**服务 2: analyze_pages**
- 类型: 网页内容深度分析
- 能力: 内容提取、结构化解析、质量评估、相关性分析
- 使用场景:
  - 搜索结果深度分析
  - 信息验证
  - 关键内容提取
- 配置掌握: 分析深度、提取字段、质量阈值
- 最佳实践:
  - 对重要页面进行深度分析
  - 提取结构化信息
  - 评估内容质量和可靠性

**服务 3: download_resources**
- 类型: 资源下载管理
- 能力: 图片/PDF/文档下载、批量处理、文件管理
- 使用场景:
  - 重要资源保存
  - 证据材料收集
  - 离线分析准备
- 配置掌握: 下载目录、文件类型、大小控制、安全检查
- 最佳实践:
  - 下载高价值的图片和文档
  - 合理组织文件结构
  - 记录资源来源信息

**服务 4: save_screenshot**
- 类型: 网页截图服务
- 能力: 网页状态保存、视觉证据收集、批量处理
- 使用场景:
  - 重要页面状态保存
  - 视觉证据收集
  - 页面变化记录
- 配置掌握: 截图尺寸、质量设置、保存路径、命名规则
- 最佳实践:
  - 保存关键证据页面
  - 截取完整或关键区域
  - 使用清晰的命名规则

**服务 5: run_browser_task**
- 类型: AI 代理型任务交付（重型工具）
- 能力: 复杂交互搜索、Google 搜索引擎、智能任务执行
- 使用场景:
  - 复杂搜索任务
  - 其他工具效果不佳时
  - 需要探索性搜索时
- 特点: 开销大、工作链路长、效果可能更好
- 触发条件:
  - **保底策略**: 其他工具结果不理想时使用
  - **主动探索**: 可以一开始就使用，探索初步方向和路径
  - **随机性**: 赋予一定的随机性和自由度，可以偶尔尝试
- 最佳实践: 谨慎使用，给予明确的任务描述

**工具组合策略**:
- 根据任务特点选择工具组合
- 先广度后深度（先 searxng_search，再 analyze_pages）
- 重要内容必须保存（download_resources + save_screenshot）
- 灵活应变，不拘泥于固定流程

#### 3. 研究成果文件标准 (Research Output Standards) 📄 极其重要!

**核心原则**: 重内容轻格式，立足事实和原文

**文件内容要求**:

**必须包含的内容**:
1. **重要事实的原文** ⭐ 最重要!
   - 不能过度总结导致细节丢失
   - 关键段落必须完整保留
   - 附带来源链接和时间戳

2. **汇总报告**（有实际内容）
   - 对收集内容的整理和分析
   - 不能只是标题框架
   - 包含自己的观察和发现

3. **关键数据和事实**
   - 数字、日期、人名、地名等
   - 结构化呈现
   - 标注来源

4. **发现的资源**
   - 图片的文件路径
   - 下载的文档路径
   - 截图的保存位置
   - 重要链接清单

**禁止的内容**:
- ❌ 过多的格式化标题（标题层级不超过3层）
- ❌ 无实质内容的总结
- ❌ 精致的 markdown 格式但内容空洞
- ❌ 过度概括导致信息丢失

**内容密度标准**:
- 至少 80% 应该是实际内容（原文、数据、分析）
- 最多 20% 是结构性元素（标题、列表框架等）

**文件组织建议**:
```
research_results/
  └── {task_id}/
      ├── report.md          # 主报告（原文+汇总）
      ├── images/            # 下载的图片
      ├── downloads/         # 下载的文档
      ├── screenshots/       # 网页截图
      └── sources.md         # 来源清单
```

#### 4. 返回规范
**只返回文件路径**:
- 不返回详细内容（避免 token 浪费）
- 提供清晰的文件路径
- 可以简短说明文件包含的内容类型

**返回示例**:
```
搜索任务已完成，研究成果已保存：
- 主报告: /research_results/task_001/report.md
- 图片资源: /research_results/task_001/images/ (12张)
- 来源清单: /research_results/task_001/sources.md
```

## 第三层：写作综合子智能体 (Writing Synthesis Subagent)

### 核心职责
- 综合所有搜索成果
- 生成最终用户报告
- 分步生成高质量内容
- 迭代修改和完善

### 关键特征与要求

#### 1. 核心原则
**基于事实，严禁编造**:
- 所有内容必须有来源依据
- 不能凭空创造信息
- 不确定的内容必须明确标注

**内容丰富，禁止空洞**:
- 必须有实质性内容
- 避免套话和废话
- 信息密度要高

#### 2. 工作流程 (分步生成策略) 🔄 核心创新!

**原则**: 禁止一次性生成全部内容（效果差）

**推荐流程**:

**Step 1: 阅读和理解**
- 读取主系统指定的所有文件
- 理解每个文件的内容和价值
- 识别可用的资源（图片、文档等）

**Step 2: 自主规划章节结构**
- 根据内容和用户需求，规划合理的章节结构
- 确定每个章节的主题和内容来源
- 设计章节之间的逻辑关系

**Step 3: 逐章节生成**
- 一次生成一个章节
- 每个章节都要内容充实
- 充分利用图片、引用等丰富内容
- 将每个章节保存为独立文件

**Step 4: 合并文件**
- 使用命令将所有章节合并成完整报告
- 确保章节之间的连贯性

**Step 5: 整体审阅**
- 阅读完整报告
- 检查逻辑连贯性
- 识别需要改进的地方

**Step 6: 最终修改**
- 根据审阅结果修改
- 完善细节
- 确保质量

**Step 7: 返回结果**
- 只返回最终文件路径
- 不返回详细内容

#### 3. 报告标准 (Final Report Standards) 📊

**定位**: 直接面向用户的最终答案

**核心要求**:
- 能够清晰、完整地解决用户的问题
- 提供足够多的信息
- 图文并茂，内容丰富
- 真实、美观、专业

**内容要求**:

**信息完整性**:
- 回答用户的所有关键问题
- 涵盖需求的各个维度
- 提供足够的细节和背景

**图文整合**:
- 充分利用下载的图片（看起来更美观、真实）
- 使用 markdown 正确引用图片
- 图片要有说明和来源标注

**资源引用**:
- 引用过程中产生的研究内容
- 链接到原始文件和资源
- 提供来源清单

**内容组织**:
- 清晰的结构和逻辑
- 重点突出
- 易于阅读和理解

**典型报告结构建议**（可根据实际调整）:
```markdown
# [报告标题]

## 执行摘要
- 核心发现
- 关键结论

## 主要发现
### [主题1]
- 详细内容
- 图片展示
- 数据支撑

### [主题2]
...

## 深度分析
- 综合分析
- 关联发现
- 趋势洞察

## 支撑材料
- 关键引用
- 数据来源
- 图片资源清单

## 信息来源
- 完整的来源列表
- 可靠性评估
```

#### 4. Markdown 引用能力
**图片引用**:
```markdown
![图片说明](./images/filename.png)
来源: [链接](url)
```

**文件引用**:
```markdown
详细数据请参考: [文件名](./downloads/data.pdf)
```

**原始内容引用**:
```markdown
> 原文摘录...
>
> —— 来源: [标题](链接), 2024-01-01
```

#### 5. 迭代修改
**支持多次调用**:
- 主系统可以多次调用进行修改
- 每次调用都要明确修改要求
- 基于之前版本进行增量修改

**修改指令示例**:
```
请修改报告 /reports/final_v1.md:
1. 读取文件内容
2. 在第二章增加关于XXX的内容（来源: /research_results/task_005/report.md）
3. 优化第三章的图片展示
4. 保存为 /reports/final_v2.md
```

## 系统级要求

### 1. 文件管理规范
**命名规范**:
- 使用清晰、描述性的文件名
- 包含时间戳或版本号
- 避免特殊字符

**目录结构**:
```
project_root/
  ├── requirements/          # 用户需求文档
  ├── research_results/      # 搜索成果
  │   └── {task_id}/
  │       ├── report.md
  │       ├── images/
  │       ├── downloads/
  │       └── screenshots/
  ├── reports/               # 最终报告
  │   ├── draft_v1.md
  │   ├── draft_v2.md
  │   └── final.md
  └── logs/                  # 过程日志（可选）
```

### 2. 质量控制
**信息可靠性**:
- 优先使用权威来源
- 交叉验证关键信息
- 标注信息的可信度

**完整性检查**:
- 确保所有关键问题都有答案
- 识别信息缺口
- 必要时补充搜索

**一致性保证**:
- 避免自相矛盾的信息
- 统一术语和表述
- 保持叙事连贯

### 3. 效率优化
**并行执行**:
- 充分利用5个并发位
- 独立任务并行执行
- 减少等待时间

**避免重复**:
- 已搜索的内容不重复搜索
- 复用已有资源
- 去重处理

**动态调整**:
- 根据结果质量调整策略
- 及时止损无效路径
- 灵活应变

## 成功标准

### 用户满意度
- 用户的问题得到完整、准确的回答
- 报告内容丰富、易读、专业
- 提供的信息有价值、可操作

### 系统性能
- 搜索覆盖面广
- 信息质量高
- 执行效率高
- 资源利用合理

### 创新性
- 能够发现非常规的信息源
- 提供独特的洞察
- 展现强大的思辨能力

---

## 特殊场景处理

### 信息不足
- 明确告知用户信息限制
- 说明已尝试的搜索路径
- 建议可能的补充方向

### 信息矛盾
- 呈现不同观点
- 分析矛盾原因
- 评估各方可信度

### 敏感内容
- 客观、中立呈现
- 标注内容性质
- 遵守伦理规范

---

## 总结

这是一个高度智能化、灵活性强、注重质量的开源情报搜索系统。三层智能体各司其职，协同工作，最终为用户提供高质量、内容丰富的情报报告。

**核心特点**:
- 主系统：战略规划者、思辨大师
- 搜索子智能体：战术执行者、工具专家
- 写作子智能体：内容综合者、报告作家

**核心价值**:
- 将复杂的情报搜索任务自动化
- 保证信息的全面性和准确性
- 生成专业、美观、实用的报告
- 解放用户的时间和精力
